1. システム概要・アーキテクチャ

目的

ボート競技大会（全日本マスターズレガッタ 2026）のレース結果をリアルタイムで観客・参加者にWebブラウザから提供する静的速報サイト。計測スタッフがCSVをアップロードするだけで、エンジニア不在でも自動更新される仕組みを実現する。

コンポーネント一覧

コンポーネント技術・サービス役割費用

ホスティングCloudflare Pages静的ファイル配信・CDN・HTTPS無料
ソースコード管理GitHub（RYUIYAMADA/masters-regatta-2026）コード・JSONデータの保管とバージョン管理無料
CSV監視・変換Google Apps Script（GAS）DriveのCSVをJSONに変換しGitHubへPush無料（制限あり）
CSV投入先Google Drive（共有フォルダ）計測スタッフがCSVをアップロードする作業領域無料

設計方針

  • サーバーレス・フルスタティック: バックエンドサーバーなし。データはすべてGitHubリポジトリ内のJSONファイルとして管理
  • ゼロランニングコスト: すべてのサービスを無料枠で運用
  • エンジニア不在での運用: スタッフがCSVをDriveにアップするだけで更新。GUIツール・コマンドライン操作不要
  • 同時アクセス耐性: Cloudflare CDNによる配信のためスパイクアクセスに強い。動的サーバーなし

3. データフロー詳細

2.1 CSVファイル仕様

ファイル命名規則

正規表現: /^(?:\d{8}_\d{6}_)?R(\d{3})_(.+)\.csv$/i

形式例備考
推奨形式R001_500m.csvレース番号3桁ゼロ埋め必須
旧形式（後方互換）20260607_070000_R001_500m.csvRowingTimerWebが自動付与する日時プレフィクス
⚠️
ファイル名に含まれる計測ポイント（500m / 1000m）は、アップロード先フォルダと一致していることをGASが検証する。不一致ファイルはスキップされる（ログに記録）。

CSVフォーマット（RowingTimerWeb出力）

# ヘッダー行（必須）
measurement_point,lane,lap_index,time_ms,formatted,race_no,tie_group,photo_flag,note
# データ行の例
500m,1,1,112834,1:52.834,1,,,
500m,2,1,113201,1:53.201,1,,,
カラム型内容必須
measurement_pointstring計測ポイント名（500m / 1000m）○
laneintレーン番号（1〜）○
lap_indexint周回インデックス（通常1）○
time_msint計測タイム（ミリ秒）○
formattedstringフォーマット済みタイム（GASは再計算するため無視）—
race_nointレース番号（ファイル名と一致させること）—
tie_groupstring同着グループID（同着の場合に同じ値を設定）—
photo_flagbool写真判定フラグ（true / 1）—
notestring備考（DNS / DNF など）—
ℹ️
GASはCSVのformatted列を無視し、time_msから独自に再計算（M:SS.ss形式、センチ秒2桁統一）。BOM（UTF-8 BOM）は自動除去。

スケジュール・エントリーCSV（大会前に1回投入）

ファイル投入先主要カラム
schedule.csv master/ race_no, event_code, event_name, category, age_group, round, date, time
entries.csv master/ race_no, lane, team_name, crew_name, athlete_names

2.2 GAS処理フロー（gas/Code.gs）

トリガー設定

  • 種別: 時間ベーストリガー（Time-driven）
  • 間隔: 2分ごと（everyMinutes(2)）
  • エントリ関数: onTrigger()
  • GAS無料枠の制約: 実行時間 90分/日（1実行あたり最大6分）
  • 自己制限: 実行時間が 4分 を超えると自動停止（CONFIG.maxExecutionMs = 4 * 60 * 1000）

onTrigger() の処理順序

  1. 二重実行防止ロック（LockService.getScriptLock()、5秒待機）
  2. GitHub API レート制限フラグを確認（制限中は15分スキップ）
  3. processPendingCSVs() — CSVの監視・処理（後述）
  4. updateTriggerHeartbeat_() — master.jsonのlast_trigger_atを更新（管理画面の死活監視用）

processPendingCSVs() の処理詳細

  1. Drive の race_csv/500m/ と race_csv/1000m/ を走査
  2. ファイル名を正規表現でマッチ → raceNo と計測ポイントを抽出
  3. 同一レース・同一ポイントで複数ファイルがある場合: 旧形式はファイル名辞書順、新形式はDrive更新日時で最新を採用
  4. 全計測ポイント（500m + 1000m）が揃ったレースのみ処理
  5. buildAndPushRaceJSON() → pushToGitHub() → CSVを processed/ へ移動

buildRaceJSON() の処理詳細

  • 各CSVをパースしてレーンIDでマージ
  • フィニッシュタイム（lastPoint）でソート → 順位付け
  • 同着判定: tie_groupが同じ非空文字列のとき同順位
  • フィニッシュなし（DNF）: status: 'dnf'、順位なし
  • splitタイム: lastPoint.time_ms - firstPoint.time_ms
  • タイムフォーマット: M:SS.cc（cc = センチ秒2桁）

GitHub Push（pushToGitHub）

  • 使用API: GitHub Contents API（PUT /repos/{owner}/{repo}/contents/{path}）
  • 既存ファイルの場合: 先にGETでSHAを取得して更新
  • コミットメッセージ: Update data/results/race_XXX.json [GAS auto-push]
  • 認証: GitHub Personal Access Token（スクリプトプロパティに保存）
⚠️
GitHub Personal Access TokenはGASスクリプトプロパティ（暗号化保存）に保管。コードにハードコードしてはならない。トークンスコープ: repo（Contents APIへの書き込みに必要）。

エラー処理

  • GitHub API 429: API_RATE_LIMITEDフラグを立て15分間スキップ
  • 個別レースの処理エラー: スクリプトプロパティ LAST_ERROR に記録し、他のレース処理は継続
  • ロック取得失敗: 即時returnしてスキップ（ログのみ）

3.2 データ取得・更新ロジック

初期ロード（loadAll）

  1. data/master.json をfetch（失敗時はユーザーフレンドリーなエラー表示）
  2. スケルトンUIを先行表示（レイアウトシフト抑制）
  3. 全レースの data/results/race_XXX.json を並列fetch（Promise.allSettled）
  4. 404は正常系（結果未投入）としてスキップ
  5. 結果をキャッシュ（resultsCache[race_no]）後にUI全描画

自動更新

  • 間隔: 120秒（CONFIG.REFRESH_INTERVAL = 120000）
  • 多重実行防止: isUpdatingフラグ
  • 更新検知: 前回のキャッシュになかった race_no を「新規更新」として検知 → トーストで通知
  • オフライン検知: navigator.onLine + online/offlineイベントでステータスバーに表示
  • 読み込み遅延検知: 8秒経過で「インターネット接続をご確認ください」メッセージを自動表示

4. インフラ・セキュリティ

  • デプロイ: GitHub mainブランチへのPush時に自動デプロイ（平均30〜60秒）
  • CDN: Cloudflareグローバルエッジネットワークで配信
  • HTTPS: 自動SSL証明書（Let's Encrypt）

// 概算ロジック
const elapsedMin = (now - periodStart) / 60000;

const estimatedUsed = estimatedRuns * 0.5;           // 1回あたり約0.5分と仮定

7. 既知の制約・リスク

設計上の制約

項目制約内容対策・許容判断
GAS実行時間 無料枠90分/日。約180回実行で枯渇 1実行あたり最大0.5〜1分程度。大会1日あたり約50〜80レースなら問題なし。管理画面で残量警告表示

Cloudflareの無料プランの制限 ビルド月500回まで（GitHub Push毎にデプロイ） 100レース × 2回Push = 200回。余裕あり
管理者画面の認証 URL知識のみによるアクセス制御（HTTP認証なし） 管理画面は読み取り専用のため機密性は低い。URLを関係者のみに共有

8. セットアップ手順（新大会向け）

  1. GitHubでリポジトリを作成し、このフォルダをpush
  2. Cloudflare Pagesでデプロイ設定（GitHubリポジトリを選択 → フレームワーク: なし → ビルドコマンド: なし → 出力: /）
  3. Google DriveにCSV投入用フォルダを作成:
    • （大会フォルダ）/race_csv/500m/
    • （大会フォルダ）/race_csv/1000m/
    • （大会フォルダ）/master/
    • （大会フォルダ）/processed/（GASが自動作成）
  4. GASプロジェクトを作成し Code.gs をコピー
  5. GASコード上部にDriveフォルダIDとGitHub Tokenを入力 → saveSetup()を実行 → 入力値を削除
  6. GASで setupAll() を実行（フォルダ構造の確認・テスト）
  7. GASトリガーを設定: onTrigger、時間ベース、2分ごと
  8. スケジュール・エントリーCSVを作成して master/ フォルダに投入 → GASの runImportMaster() を実行
  9. 管理画面でパイプライン状態が「正常」になることを確認
  10. テストCSVを race_csv/500m/ と race_csv/1000m/ に投入し、2〜3分後にサイトに反映されることを確認
✅
前日セットアップが完了していれば、当日は計測スタッフがCSVをアップするだけでサイトは自動更新される。エンジニアの当日常駐は不要（リモートバックアップ推奨）。

9. 不完全な点・リスク・改善提案（レビュー対象）

🔴 高リスク（本番障害につながる可能性）

#問題影響改善提案
R-1
解決済		 GAS実行時間の枯渇
無料枠90分/日。当初は懸念していたが実際の使用量を試算したところ問題なし。		 実測値ベース: ハートビートのみ実行≈5秒/回、CSV処理≈20秒/回。10時間稼働・1時間3レース処理で約32〜40分/日の消費。90分上限に対して余裕がある。
運用ルール: トリガーは大会前日から有効化し、最終日終了後に停止（チェックリストに明記）。管理画面でリアルタイム残量をプログレスバー表示するよう改善済み。		 現状の運用ルールで対処可能。改修不要。
R-2 GitHub Personal Access Tokenの有効期限
トークンに有効期限を設定している場合、大会直前に失効する可能性		 GASがGitHub APIにアクセスできなくなり、結果が反映されない セットアップ時にexpiration: neverのトークンを使用する、または大会前日にトークン有効期限を確認するチェックリスト項目を追加
R-3 計測ポイントが揃わない場合に永続的にスキップ
片方のCSVが投入されないと、そのレースは永遠に処理されない		 結果の欠損 500mのみ / 1000mのみのコースの場合も扱えるよう、master.jsonの種目ごとにcourse_typeを設定する設計が必要。現状は全種目が同一計測ポイントセットと仮定している
R-4 master.jsonの整合性チェックなし
schedule.csvとentries.csvのrace_noが一致しない場合にGASはエラーを出さない		 エントリー情報が欠損した状態で結果が表示される（選手名が空欄） importMasterData()実行時にrace_noの整合性チェックを追加し、不一致をGASログに警告

🟡 中リスク（運用で回避可能だが設計上の課題）

#問題影響改善提案
M-1 CSVファイル名のケース非感知の不統一
正規表現パターンに /i フラグがあるが、フォルダ内一致チェック（filePoint !== point）は大文字小文字を区別する		 「R001_500M.csv」（Mが大文字）はパターンマッチは通るが、フォルダ一致チェックで失敗してスキップされる filePoint.toLowerCase() !== point.toLowerCase() に統一
M-2 管理者画面に認証なし
URLが漏洩した場合、誰でも管理画面にアクセス可能		 大会スケジュール・エントリー情報が閲覧される Cloudflare Access（無料枠あり）でメール認証を追加するか、Cloudflare Pagesの「Protected Pages」機能（有料）を利用する
M-3 Content-Security-Policyに unsafe-inline を許可
インラインJSが多数あるため止むを得ないが、XSSリスクが増加する		 外部攻撃者がCSPをバイパスしやすくなる インラインJSをすべて外部ファイルに移動し、script-src 'self'のみに変更する。またはCSP nonceを導入する
M-4 自動更新間隔が120秒に固定
画面を常時表示していても、更新タイミングにランダム性がない		 全ユーザーが同時にfetchするためサーバー（Cloudflare）への同時リクエストが集中する可能性 初回ロード時にランダムなオフセットを加算する: REFRESH_INTERVAL + Math.random() * 30000
M-5 GASハートビートがGitHub APIの無駄なPushを発生させる
CSVがなくてもトリガーごとにmaster.jsonをPushする		 Cloudflare Pagesのビルド回数消費（月500回上限）。100レースの大会で最大数十回の余分なデプロイが発生 ハートビートのみの更新はGitHub Contents APIではなく、GASスクリプトプロパティへの保存で代替し、管理画面はGASのwebapp経由で取得する設計に変更
M-6 resultsCache のメモリ増大
長時間閲覧しているとresultsCacheにレース数分のJSONが蓄積される		 古いブラウザやメモリの少ないデバイスでページが重くなる可能性 レース数が多い大会（100件以上）では問題になりうる。LRUキャッシュ（最新50件のみ保持）の導入を検討

🟢 低リスク（改善でUXが向上する）

#問題改善提案
L-1 エラー時のリトライロジックなし
fetchが失敗した場合、次の自動更新（120秒後）まで待つしかない		 指数バックオフリトライ（1回目失敗→10秒後、2回目→30秒後）を実装する
L-2 data/results/のファイルが削除されても検知できない
誤ってGitHub上のJSONを削除した場合、そのレースの結果がサイトから消える		 GASのrunNow()を再実行すれば復元可能。CSVがprocessed/に移動済みのため再処理はできない点を引き継ぎ書に明記すべき
L-3 タイムゾーンが明示的に設定されていない
GASのDate()はスクリプトのタイムゾーン設定（appsscript.jsonのtimeZone）に依存		 appsscript.jsonの"timeZone": "Asia/Tokyo"を確認済みであれば問題なし。設定漏れに注意
L-4 テスト自動化なし
e2e_test.pyはあるが、GASのユニットテストは存在しない		 GASのローカルテスト環境（clasp + jest）の導入、またはGASWebApp経由でのAPIテストを検討
L-5 processed/フォルダの定期清掃フローなし
大会ごとにprocessed/にCSVが蓄積される		 大会終了後の運用手順書（site-checklist.htmlの終了後チェック）に「processed/を空にする」手順を明記する
ℹ️
上記のうち、本番運用前に対処すべき最優先項目は R-1・R-2・M-1 の3件。特にR-2（トークン有効期限）は大会前日チェックリストへの追加で即対応可能。
